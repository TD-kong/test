# 1. 概要

HULFTは、サーバ間でファイル転送を行うミドルウェアであり、企業システムにおいてバッチ連携や外部システム連携に広く利用される。

運用保守では主に以下の観点で利用する。

- HULFTサービス状態確認
- 配信要求実施
- 転送結果確認
- 異常時の再送
- 滞留確認
- ログ調査

---

# 2. HULFTサービス操作

## サービス起動

```bash
utlsvc -start
```

用途：
- HULFTサービス起動

実務用途：
- OS再起動後
- 障害復旧後

---

## サービス停止

```bash
utlsvc -stop
```

用途：
- HULFTサービス停止

注意：
- 転送中ファイルがないことを確認すること

---

## サービス状態確認

```bash
utlsvc -status
```

用途：
- HULFTサービス稼働確認

確認ポイント：
- 正常起動状態か
- プロセス異常終了していないか

---

# 3. 配信操作コマンド

## 配信要求

```bash
utlsend -f ファイルID
```

例：

```bash
utlsend -f SALES_DATA
```

用途：
- 指定ファイルIDの配信要求

実務用途：
- 手動再送
- 緊急配信

---

## 再送

```bash
utlsend -f ファイルID
```

運用観点：
1. 異常原因確認
2. 相手先稼働確認
3. 再送実施

注意：
- 無条件再送禁止
- 二重連携リスク確認

---

# 4. 状態確認コマンド

## 集配信状態確認

```bash
utlobs
```

用途：
- 集信／配信状態確認

確認観点：
- 処理中
- 異常終了
- 滞留
- 正常終了

実務メモ：
- 障害時最初に確認することが多い

---

## 転送状況確認

```bash
utllist
```

用途：
- 配信履歴確認

確認ポイント：
- 正常終了有無
- エラーコード
- 相手ホスト

# 5. 障害時によく使う操作

## ケース1：転送失敗

### 手順

1. HULFTサービス確認

```bash
utlsvc -status
```

2. 状態確認

```bash
utlobs
```

3. ログ確認

4. 相手先稼働確認

5. 再送

```bash
utlsend -f FILE_ID
```

---

## ケース2：滞留

確認観点：

- 相手サーバ停止
- 通信障害
- ポート疎通
- 転送ファイルロック

---

## ケース3：サービスダウン

```bash
utlsvc -stop
utlsvc -start
```

確認：

```bash
utlsvc -status
```

# 6. ログ確認

## HULFTログ確認

環境によって異なるが、一般例：

```bash
cd /usr/local/hulft/log
ls -ltr
```

確認ポイント：

- 通信エラー
- 相手先拒否
- ファイルロック
- ディスク容量不足

Linuxログ確認例：

```bash
grep ERROR *.log
```

```bash
tail -100 logfile.log
```

# 7. JP1連携例

JP1ジョブからHULFT送信：

```bash
#!/bin/sh

utlsend -f SALES_DATA
RET=$?

if [ $RET -ne 0 ]; then
  exit 1
fi

exit 0
```

運用ポイント：

- 戻り値判定必須
- 異常時JP1異常終了へ連携
- ログ出力推奨

# 8. シェル実装例

## 手動再送スクリプト

```bash
#!/bin/sh

FILE_ID=$1

if [ -z "$FILE_ID" ]; then
  echo "FILE_ID未指定"
  exit 1
fi

utlsend -f $FILE_ID
RET=$?

if [ $RET -eq 0 ]; then
  echo "送信成功"
else
  echo "送信失敗"
fi

exit $RET
```

# 9. 終了コード観点

|終了コード|意味|対応|
|---|---|---|
|0|正常終了|処理継続|
|0以外|異常終了|ログ確認・再送検討|

注意：

環境や設定によって戻り値定義が異なる場合があるため、運用設計書に準拠すること。
