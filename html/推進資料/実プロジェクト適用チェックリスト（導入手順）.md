# 実プロジェクト適用チェックリスト（導入手順・サンプル）

## 1. 目的
本書は、標準規約・テンプレート群（設計・テスト・運用・監査）を実プロジェクトへ適用するための導入手順およびチェックポイントを定義する。

---

## 2. 適用前提条件

以下が揃っていることを前提とする。

- プロジェクト標準規約が策定済み
- 各種テンプレート（設計・テスト・運用）が存在
- リポジトリ（Git等）または共有基盤が利用可能
- レビュー体制（PM・QA・開発リーダ）が確立

---

## 3. 導入フェーズ

| フェーズ | 内容 |
|----------|------|
| Phase 0 | 事前準備（標準適用可否判断） |
| Phase 1 | プロジェクト初期設定 |
| Phase 2 | 成果物テンプレ適用 |
| Phase 3 | 開発プロセス統制開始 |
| Phase 4 | 運用・監視統合 |
| Phase 5 | 定着化・監査対応 |

---

## 4. Phase 0：事前準備チェック

- [ ] プロジェクト規模が標準適用対象か確認
- [ ] ステークホルダー合意取得済み
- [ ] 標準逸脱の許容範囲明確化
- [ ] 教育計画策定済み

---

## 5. Phase 1：プロジェクト初期設定

- [ ] フォルダ構成テンプレ適用
- [ ] リポジトリ作成（ブランチ戦略含む）
- [ ] 命名規約の周知
- [ ] 標準ドキュメント配置

### チェック対象

- プロジェクト標準規約
- トレーサビリティマトリクス
- 変更管理プロセス

---

## 6. Phase 2：成果物テンプレ適用

- [ ] 要件定義テンプレ適用
- [ ] 設計書テンプレ適用
- [ ] テスト仕様書テンプレ適用
- [ ] 運用設計テンプレ適用

### 必須統制項目

- 成果物ID付与ルール統一
- 版管理ルール適用
- レビュー承認フロー有効化

---

## 7. Phase 3：開発プロセス統制

- [ ] 設計レビューゲート運用開始
- [ ] コーディング規約適用
- [ ] 単体テスト必須化
- [ ] CI/CDまたはリリース手順統一

### チェックポイント

- 未レビューコードのマージ禁止
- 本番直接変更禁止
- テスト未完了リリース禁止

---

## 8. Phase 4：運用・監視統合

- [ ] 運用Runbook配置
- [ ] 監視設計導入
- [ ] KPIダッシュボード構築
- [ ] 障害対応フロー適用

### 監視対象

- 稼働率
- 応答時間
- エラー率
- バッチ成功率

---

## 9. Phase 5：定着化・監査対応

- [ ] 監査チェックリスト適用
- [ ] 証跡管理開始
- [ ] RCAプロセス運用
- [ ] 定期レビュー実施

---

## 10. 適用成功判定基準

| 指標 | 基準 |
|------|------|
| ドキュメント統一率 | 100% |
| トレーサビリティ | 100% |
| テストカバレッジ | 80%以上 |
| 障害対応時間 | MTTR 30分以内 |

---

## 11. よくある導入失敗パターン

| 失敗 | 原因 | 対策 |
|------|------|------|
| テンプレ未使用 | 現場負荷 | 教育不足 |
| 形骸化 | 運用未整備 | KPI未設定 |
| 例外乱発 | 標準不明確 | 承認フロー不足 |

---

## 12. 定着化施策

- 初期2週間は強制適用期間
- レビューで逸脱検知
- KPIで可視化
- 月次監査で改善

---

## 13. 運用ルール

- 標準逸脱はチケット化必須
- 例外は期限付き承認
- 恒久対策必須

---

## 14. 付録

- プロジェクト標準規約
- 監査チェックリスト
- KPIダッシュボード設計
- 障害報告書（RCA）
